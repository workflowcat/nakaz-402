Беремо реальний приклад: **наказ № 153 від 27.02.2025**
([z0329-25](https://zakon.rada.gov.ua/laws/show/z0329-25)), яким доповнили
главу 1 розділу I новим пунктом 1.4 — «електронна інформаційна взаємодія
через систему «Трембіта»».

У класичному режимі (як зараз на rada.gov.ua) для розуміння цієї зміни треба:

1. знайти наказ № 153 у переліку амендментів;
2. відкрити його окремим документом;
3. зрозуміти, куди саме він вклинюється — «доповнено главу 1 пунктом 1.4»;
4. подивитись на текст 402-го, переконатись, що дійсно є місце для нового пункту;
5. подумки приклеїти текст 1.4 до тексту 1.3.

У git-режимі це виглядає так:

---

## Крок 1. У ваш Slack / Telegram приходить наказ № 153

Юрист, який слідкує за нормативкою, помічає публікацію на rada.gov.ua.
Створює задачу.

## Крок 2. Гілка під амендмент

```bash
git checkout -b amendment/153-2025-02-27
```

Конвенція: одна гілка = один наказ-зміна.

## Крок 3. Правки у файлах

У `polozhennia/01-osnovy-organizatsii/01-zagalni-polozhennia.md`:

1. **У frontmatter** додаємо запис у `amended_by`:

```yaml
amended_by:
  # ... попередні
  - { date: 2025-02-27, order: z0329-25, op: insert, scope: "пункт 1.4 — новий" }
```

2. **У тексті** дописуємо новий пункт 1.4:

```markdown
<a id="p1.4"></a>
## 1.4.

Електронна інформаційна взаємодія, передбачена цим Положенням, здійснюється
засобами системи електронної взаємодії...

> *{Главу 1 розділу I доповнено новим пунктом 1.4 згідно з Наказом № 153 від
> 27.02.2025 ([z0329-25](https://zakon.rada.gov.ua/laws/show/z0329-25#n22))}*
```

3. **Оновлюємо `last_amended`** (необов'язково для лінтера, але корисно для людей).

## Крок 4. Журнал амендментів

У [`meta/amendments.yaml`](../meta/amendments.yaml) додаємо запис:

```yaml
- order: "153"
  signed_at: 2025-02-27
  registration: z0329-25
  summary: >-
    Доповнено главу 1 розділу I новим пунктом 1.4
    (електронна інформаційна взаємодія через «Трембіту»).
  affects:
    - id: polozhennia.r1.gl1.p1.4
      op: insert
```

## Крок 5. Журнал для людей

У [`CHANGELOG.md`](../CHANGELOG.md) додаємо запис у розділ «2025»:

```markdown
### Наказ № 153 від 2025-02-27 — [z0329-25](https://zakon.rada.gov.ua/laws/show/z0329-25)
**Новий пункт 1.4** глави 1 розділу I: електронна інформаційна взаємодія
здійснюється через систему «Трембіта».
```

> 💡 Якщо ми зробимо генератор з `amendments.yaml` → `CHANGELOG.md`, цей
> крок можна автоматизувати.

## Крок 6. Pull Request

```bash
git add .
git commit -m "amendment: наказ № 153 від 27.02.2025 — новий п. 1.4 (z0329-25)"
git push -u origin amendment/153-2025-02-27
gh pr create --title "Amendment: Наказ № 153 від 27.02.2025 (z0329-25)" \
             --body "Доповнено главу 1 розділу I новим пунктом 1.4..."
```

Шаблон PR description (TODO: додати в `.github/PULL_REQUEST_TEMPLATE.md`):

```markdown
## Наказ-зміна
- **Номер:** 153
- **Дата підписання:** 2025-02-27
- **Реєстрація в Мін'юсті:** [z0329-25](https://zakon.rada.gov.ua/laws/show/z0329-25)

## Що змінено
- доповнено главу 1 розділу I новим пунктом 1.4

## Зачеплені файли
- `polozhennia/01-osnovy-organizatsii/01-zagalni-polozhennia.md` (новий п. 1.4)
- `meta/amendments.yaml` (запис про наказ)
- `CHANGELOG.md`

## Чек-ліст
- [x] У всі змінені файли `polozhennia/` додано запис в `amended_by`.
- [x] Додано запис у `meta/amendments.yaml`.
- [x] Перевірено за офіційним текстом на rada.gov.ua.
- [ ] Лінтер frontmatter (TODO).
```

## Крок 7. Ревью

Ревьюер відкриває diff і бачить:

- у `01-zagalni-polozhennia.md` — рівно один новий пункт `## 1.4.` і один
  новий запис у `amended_by`;
- у `amendments.yaml` — рівно один новий запис з `affects: [...p1.4]`;
- у CHANGELOG — людський опис.

Звіряє з офіційним текстом на rada.gov.ua (один клік за лінком). Approve.

## Крок 8. Мерж і теги

```bash
gh pr merge --squash
git checkout main && git pull
git tag redaction/2025-02-27 -m "Редакція станом на 2025-02-27 (наказ № 153)"
git push --tags
```

Все. Тепер:

- `git checkout redaction/2025-02-27` — стан Положення на цю дату.
- `git log -- polozhennia/01-osnovy-organizatsii/01-zagalni-polozhennia.md`
  — повна історія цього пункту, кожен коміт = окремий амендмент.
- На GitHub Pages автоматично оновився сайт.

## Що дає такий процес

| Фіча                                    | Кому корисно            |
| --------------------------------------- | ----------------------- |
| Чіткий diff кожного амендменту          | юрист, ревьюер          |
| Історія по кожному пункту               | методолог, дослідник    |
| Машино-читабельний `amendments.yaml`    | data engineer, аналітик |
| Сайт з пошуком і нав-меню               | користувач (медик, командир) |
| Тегована поточна редакція               | архівіст, аудитор       |
| Можливість порівняти редакції на дати   | юрист, історик законодавства |

І найголовніше — **узгодження нового амендменту з ~100-сторінкового
переписування цілого Положення зводиться до одного PR з diff-ом на 5—10
рядків**. Те, що користувач має на меті: швидкість погодження апдейтів.
