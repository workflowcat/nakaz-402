## Загалом

Класичний нормативно-правовий акт у тому вигляді, у якому він живе на
[rada.gov.ua](https://zakon.rada.gov.ua/) — це **один великий HTML/DOC**.
Зміни накопичуються «маркерами» у фігурних дужках всередині тексту:

> *{Пункт 1.1 — в редакції Наказу № 262 від 27.04.2024}*

Щоб зрозуміти, що було в редакції на конкретну дату — треба читати наказ
№ 262 окремо, у голові «накладати» його на текст 402-го, перевіряти, чи
були ще наступні правки в цей же пункт. Це процес ручний, і він не масштабується.

Ми переписуємо цей процес у git-стилі. Замість:

- **великого файла** — багато маленьких;
- **маркерів у тексті** — структуровані метадані у YAML-шапці кожного файла;
- **окремих наказів-патчів** — Pull Request-и з diff-ами;
- **«поточної редакції»** — git-теги типу `redaction/2025-08-22`;
- **«знайти, де змінили п. 1.2»** — `git log -- polozhennia/.../...`.

## Архітектура

### 1. Гранулярність

Кожна **глава** Положення — окремий файл `.md`. Кожен **додаток** — окремий
файл або тека (для великих, як Розклад хвороб). Зміна одного пункту торкається
рівно одного файла; diff PR-у читається швидко.

Дрібніше — на рівні пункту — теоретично можна, але обертається 300+ файлами і
ускладнює навігацію. Поточний компроміс — глава = файл. Аналогічно роблять
проекти типу [indigo-akn](https://github.com/laws-africa/indigo).

### 2. Стабільні `id`

Кожен файл/розділ/пункт має `id` у frontmatter:

```yaml
---
id: polozhennia.r1.gl1.p1.1
type: punkt
parent: polozhennia.r1.gl1
title: "Загальні положення"
---
```

`id` НЕ міняється між редакціями. Якщо пункт скасовано — він залишається з
`status: repealed`, не видаляється. Це дає стабільні посилання з інших
документів і нормальну історію git blame.

Граматика `id` — у [meta/schema.md](../meta/schema.md).

### 3. Історія амендментів — у файлі

У frontmatter кожного пункту:

```yaml
amended_by:
  - { date: 2023-08-18, order: z1467-23, op: edit,     scope: "абзац 2" }
  - { date: 2024-04-27, order: z0616-24, op: redaction }
  - { date: 2025-08-01, order: z1167-25, op: edit }
```

Зверху, у тексті, ці правки маркуються як `> *{...}*` (звичним стилем
rada.gov.ua) — для людей-читачів. Для машин — використовуємо `amended_by`.

Дублювання навмисне: текстові маркери легко звірити з офіційним джерелом
очима, машинні поля — лінтером.

### 4. «Конфіги» — у `dodatky/`

Усе, що змінюється часто і має табличну структуру (Розклад хвороб, ТДВ,
форми довідок), винесено в `dodatky/`. Найважливіший приклад —
[`dodatky/01-rozklad-khvorob/stattia-001.yaml`](../dodatky/01-rozklad-khvorob/stattia-001.yaml):

```yaml
stattia: 1
nazva: "Туберкульоз органів дихання"
punkty:
  - id: "а"
    opys: "активний прогресуючий..."
    grafy: { I: Д, II: Д, III: Д, IV: Д }
  - id: "б"
    opys: "активний обмежений..."
    grafy: { I: В, II: В, III: Б-4, IV: В }
```

Це **той самий вміст**, що в тексті Положення, але як таблиця. Тепер:

- редагування пункту «б, графа III» = одна правка в YAML;
- лінтер скаже, якщо забули заповнити графу;
- з цього можна автоматично побудувати «калькулятор придатності».

Схема валідації — у [`dodatky/01-rozklad-khvorob/meta.yaml`](../dodatky/01-rozklad-khvorob/meta.yaml).

### 5. Журнал амендментів

[`meta/amendments.yaml`](../meta/amendments.yaml) — машино-читабельний журнал.
Кожен запис — окремий наказ-зміна:

```yaml
- order: "262"
  signed_at: 2024-04-27
  registration: z0616-24
  summary: "Велика правка. Нові редакції пп. 1.1, 1.2..."
  affects:
    - { id: polozhennia.r1.gl1.p1.1, op: redaction }
    - { id: polozhennia.r1.gl1.p1.2, op: redaction }
    - { id: polozhennia.r1.gl1.p1.3, op: insert }
```

Поле `affects` зв'язує наказ-зміну з конкретними пунктами Положення. З цих
даних можна:

- автоматично згенерувати CHANGELOG.md;
- побудувати «mini-history» для кожного пункту;
- зробити пошук типу «всі правки, що внесли в 2024»;
- порівняти редакції на дві довільні дати.

### 6. Версіонування через git

| Сутність                  | Як виражена в git                                  |
| ------------------------- | -------------------------------------------------- |
| Поточна редакція          | стан `main` гілки                                  |
| Редакція станом на дату X | `git checkout redaction/YYYY-MM-DD`                |
| Один наказ-зміна          | один PR (squash merge)                             |
| Хто змінив пункт          | `git log -- polozhennia/.../...md`                 |
| Що змінив наказ           | PR за номером або `git log --grep="z1467-23"`      |

## Що дає GitHub Pages

Сайт, який ви зараз читаєте, генерується з тих самих markdown-файлів, що
лежать у репо. Jekyll бере frontmatter + текст і будує статичні сторінки з:

- двопанельною навігацією (зміст ліворуч, контент посередині, on-page TOC праворуч);
- пошуком (`/` — фокус, повнотекстовий);
- темним і світлим режимом;
- хлібними крихтами;
- автоматичним sitemap.xml для SEO.

Деплой автоматичний — кожен push у `main` тригерить GitHub Actions, які
збирають сайт і публікують на `https://<user>.github.io/<repo>`.

## Що **не** робить цей сайт (поки)

- **Не зберігає юридично авторитетний текст.** Для рішень — джерело
  залишається rada.gov.ua.
- **Не валідує frontmatter автоматично.** TODO: GitHub Action з простим
  скриптом, що перевіряє формат `id`, наявність `source`, `status` тощо.
- **Не генерує diff-and-amend.** TODO: скрипт, що бере PR і автоматично
  перевіряє, що для кожної правки тексту в `polozhennia/` додано запис в
  `amended_by` цього ж файла.
- **Не показує редакцію на дату.** TODO: інтерактивний slider «станом на …»
  (потребує per-абзацного маркування редакцій або вичитки з git blame).

## Технологічний стек

| Шар           | Технологія                                  |
| ------------- | ------------------------------------------- |
| Файли         | Markdown + YAML frontmatter; YAML для додатків |
| Версіонування | git                                         |
| Хостинг       | GitHub                                      |
| Сайт          | Jekyll + тема `just-the-docs`               |
| Деплой        | GitHub Pages (через GitHub Actions)         |
| CI (план)     | GitHub Actions + Python/Ruby лінтер         |

Свідомо обрано найпростіший зрілий стек: Jekyll вбудований у GitHub Pages,
не потребує окремого сервера. Якщо знадобиться більше — мігрувати на MkDocs,
Docusaurus або власне React-SPA не проблема, бо вихідний контент — звичайний
Markdown.
