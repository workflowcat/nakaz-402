# Наказ МОУ № 402: гідхабізація

Спроба перевести **Наказ Міністерства оборони України № 402 від 14.08.2008**
«Про затвердження Положення про військово-лікарську експертизу в Збройних
Силах України» (зареєстровано в Мін'юсті 26.08.2008 за № 1109/15800) у формат,
з яким зручно працювати інженерам, юристам і медикам одночасно: машино-читабельний,
з історією версій по кожному пункту, з PR-флоу на зміни, з diff-ами замість
«знайдіть всі правки за останній рік».

> **Статус:** скелет / прототип. Демонстрація підходу, а не остаточна форма НПА.
> Юридично авторитетним залишається текст на [rada.gov.ua](https://zakon.rada.gov.ua/laws/show/z1109-08).

---

## Навіщо це

Наказ 402 — це **24 категорії осіб**, **5+ додатків** (Розклад хвороб, Пояснення,
ТДВ, форми довідок), і **20+ наказів-змін** з 2008 року (від наказу № 577 від 2010
до № 518 від 01.08.2025). Останнього разу його суттєво перекраяли наказом № 262
у квітні 2024 і потім ще тричі в 2025-му.

Класична проблема:

- щоб дізнатись, що змінилось, треба читати окремий наказ-зміну, шукати
  пункти за номером в основному тексті, в голові робити merge;
- кожен апдейт = переписують ціле положення цілком, бо «всі конфіги
  захардкоджені»: списки категорій, переліки органів, таблиці хвороб,
  форми довідок — усе вшито в плоский текст;
- історичну редакцію на конкретну дату відновити боляче.

Що ми міняємо:

| Класичний НПА                            | Цей репозиторій                          |
| ---------------------------------------- | ---------------------------------------- |
| Один великий .doc / HTML                 | Розрізаний по розділах і додатках        |
| Пункти ідентифікуються словами в тексті  | Кожен пункт має стабільний `id` у YAML   |
| Зміни — окремими наказами-патчами        | Зміни — PR з diff-ом, тегами на редакції |
| Списки/таблиці захардкоджені в положенні | Винесені в `dodatky/` як машинні конфіги |
| Знайти, що чіпав наказ № X — складно     | `meta/amendments.yaml` + git blame       |

## Стабільні ідентифікатори

Кожен пункт, глава, розділ і додаток має стабільний `id` у frontmatter.
Формат: `<scope>.<rN>.<glN>.<pN.N>`.

Приклади:

- `polozhennia.r1.gl1.p1.1` — Розділ I, Глава 1, пункт 1.1
- `polozhennia.r2.gl1.p1.4` — Розділ II, Глава 1, пункт 1.4
- `dodatok.1.stattia.42` — стаття 42 Розкладу хвороб
- `nakaz.p3` — пункт 3 самого наказу

Ці `id` НЕ міняються між редакціями. Якщо пункт скасовано — він залишається
з `status: repealed`, не видаляється. Це дає стабільні посилання з інших
документів і нормальну історію в git blame.

## Як читати один пункт

Зразок: [polozhennia/01-osnovy-organizatsii/01-zagalni-polozhennia.md](polozhennia/01-osnovy-organizatsii/01-zagalni-polozhennia.md)

У файлі — YAML-шапка (метадані: id, статус, історія амендментів, посилання на
офіційне джерело) + сам текст пункту. У git-історії цього файлу видно всю
історію редакцій конкретного пункту з підв'язкою до наказу, що його змінив.

## Як вносити зміни (workflow)

1. Виходить новий наказ-зміна (умовний № 999 від 01.06.2026).
2. Створюємо гілку `amendment/999-2026-06-01`.
3. Один PR = один наказ-зміна. У PR description — посилання на офіційне джерело
   наказу та коротке резюме.
4. У кожному зміненому файлі:
   - оновлюємо текст;
   - додаємо запис у блок `amended_by` у frontmatter;
   - якщо пункт втрачає чинність — ставимо `status: repealed`, текст залишаємо.
5. Додаємо запис у `meta/amendments.yaml`.
6. Після мержу — тегуємо коміт: `redaction/2026-06-01`.

Це дає:

- **«покажи редакцію станом на 14.08.2024»** = `git checkout` за тегом дати;
- **«хто і коли змінив пункт 2.5»** = `git log -- polozhennia/.../...md`;
- **«що змінив наказ № 490»** = `git log --grep="z1467-23"` або PR на GitHub.

## Що тут готового

| Файл                                                     | Стан                          |
| -------------------------------------------------------- | ----------------------------- |
| `nakaz.md`                                               | повний дослівний текст наказу |
| `polozhennia/01-osnovy-organizatsii/01-zagalni-...md`    | повний текст, із frontmatter  |
| Інші файли `polozhennia/`                                | заглушки + frontmatter        |
| `dodatky/01-rozklad-khvorob/`                            | заглушка + схема статті       |
| `dodatky/02..05`                                         | заглушки + frontmatter        |
| `meta/amendments.yaml`                                   | усі ~23 накази-зміни          |
| `meta/glossary.md`, `meta/schema.md`                     | готові                        |
| `CHANGELOG.md`                                           | людинописна версія amendments |

**Дотягнути:**

- решта пунктів Положення (велика робота — кожен абзац потребує звірки з офіційним
  текстом і розставлення `amended_by` на основі приміток у фігурних дужках);
- статті Розкладу хвороб з машино-читабельними категоріями придатності
  (показовий зразок — `dodatky/01-rozklad-khvorob/stattia-001.yaml`);
- лінтер frontmatter (валідатор `id`, обов'язкових полів, посилань);
- GitHub Action: при PR — порівнює, чи кожна правка тексту супроводжується
  записом у `amended_by`.
